# План: два сценария (обзор рынка / конкурентный анализ)

## Текущее состояние

- Оркестратор принимает `--mode market | competitors` в командной строке.
- Входной JSON: `mode: "template" | "free_form"`, `top_n`, `template` или `free_form_text`.
- Block6: два разных отчёта — `report_market` и `report_competitors` (карточки конкурентов + рекомендации).
- Режим отчёта не связан с выбором пользователя «с самого начала».

---

## Целевая логика

Пользователь **сразу** выбирает один из двух сценариев. Дальше поток и состав отчёта зависят от этого выбора.

---

## Сценарий 1: Обзор рынка

- **Выбор пользователя:** «Обзор рынка».
- **Вход:** как сейчас — два подрежима:
  - **template** — поля: types, cuisines, price_min, price_max, особенности;
  - **free_form** — свободный текст (название/описание желаемого).
- **Поток:** без изменений. Block1 → … → Block6. Top N заведений из статичной БД (по умолчанию 10).
- **Отчёт:** тот же, что сейчас в режиме `market`: анализ рынка, меню, отзывов, маркетинга, сайтов, рекомендации по нише, справочная информация. Никаких изменений в структуре отчёта.

**Изменения в коде:**
- Входной контракт: явно считать, что это «обзор рынка», например через поле `report_type: "market"` (или оставить текущий `mode` только для template/free_form).
- Оркестратор больше не принимает `--mode` из командной строки для выбора типа отчёта: тип отчёта берётся из `input_request.json` (например `report_type` из b1).

---

## Сценарий 2: Конкурентный анализ

- **Выбор пользователя:** «Конкурентный анализ».
- **Вход — одно заведение пользователя (опорное), поля:**
  - **Название** (обязательно)
  - **Адрес** (обязательно)
  - **Ссылка на Яндекс.Карты** (опционально, для отзывов и привязки)
  - **Меню** — файл в формате PDF или PNG (путь к файлу или URL)
  - **Сайт** (опционально)
- **Поток:** тот же пайплайн (Block1 → Block2 → … → Block6), но:
  - Всегда **11 мест**: 10 из нашей БД (конкуренты/похожие) + **11-е место — заведение пользователя**.
  - В финальном отчёте «крутим-вертим» именно **11-е место**: как оно смотрится на фоне 10 конкурентов, где сильнее/слабее, что улучшить.

**Изменения по блокам:**

### Block1 (relevance)

- **Новый режим входа:** `report_type: "competitive"` и объект `reference_place`: название, адрес, ссылка_яндекс_карты?, меню (путь/URL), сайт?.
- **Логика:**
  1. По заведению пользователя (название + адрес, при необходимости — Perplexity) определить тип/кухню/ценовой диапазон.
  2. По нашей БД найти **10 похожих/конкурентных** заведений (тот же поиск, что в обзоре рынка, но запрос строится из данных опорного места).
  3. Сформировать **11-ю карточку** из введённых данных пользователя (название, адрес, сайт, ссылка на меню/файл меню, ссылка Яндекс.Карты). Описание/тип/кухня/средний_чек — из Perplexity или дефолты.
  4. Выход Block1: `selected_places` = [10 из БД] + [заведение пользователя как 11-й элемент]. В каждом месте явно помечать, кто «из базы», кто «заведение пользователя» (например флаг `is_reference_place: true` для 11-го).

### Block2 (menu)

- Для мест 1–10 — без изменений (меню из CSV/ссылок).
- Для **11-го места**: источник меню — переданный пользователем **PDF или PNG** (файл или URL). Текущая логика парсинга меню (OpenRouter/Vision) остаётся, меняется только источник: не скачивание по ссылке из CSV, а чтение переданного файла или URL из `reference_place.menu_file` / `reference_place.menu_url`.

### Block3 (reviews)

- Для мест 1–10 — без изменений (матч по название+адрес с CSV, запуск парсера Яндекс.Карт).
- Для **11-го места**: если есть `reference_place.yandex_maps_link`, извлечь из ссылки **org_id** и вызвать парсер отзывов по этому org_id (без поиска по названию в Картах). Если ссылки нет — блок отзывов для 11-го места пустой или «нет данных».

### Block4 (marketing) / Block5 (tech)

- Работают с тем же списком из 11 мест. Для 11-го данные брать из `reference_place` (сайт и т.д.) и при необходимости дополнять из Perplexity/внешних источников, чтобы структура карточки была единой.

### Block6 (aggregator)

- **Убрать** отдельный режим «обзор конкурентов» в текущем виде (отдельный тип отчёта с карточками всех без акцента на одном).
- **Один тип отчёта** по сценарию:
  - Если `report_type === "market"` — генерировать отчёт как сейчас **report_market** (те же разделы и структура).
  - Если `report_type === "competitive"` — генерировать отчёт **конкурентного анализа**:
    - Использовать те же данные (b1–b5), но в отчёте явно выделить **11-е заведение** как «ваше»:
      - секции: сравнение вашего заведения с 10 конкурентами (позиционирование, меню, отзывы, маркетинг, сайт);
      - выводы и рекомендации сформулированы относительно **вашего** места: где вы сильнее, где отстаёте, что улучшить.
    - Можно переиспользовать построение карточек из текущего `report_competitors`, но промпты и структуру разделов переписать под «ваше место vs 10 конкурентов», а не «просто карточки всех».
- Оркестратор передаёт в Block6 не `args.mode`, а `report_type` из входного запроса (или из b1).

## Структура входного JSON (целевая)

### Обзор рынка

```json
{
  "report_type": "market",
  "top_n": 10,
  "source_csv": "path/to/final_blyat_v3.csv",
  "mode": "template",
  "template": {
    "types": ["ресторан"],
    "cuisines": ["русская"],
    "price_min": 2000,
    "price_max": 5000,
    "особенности": "уютная атмосфера"
  }
}
```

или

```json
{
  "report_type": "market",
  "top_n": 10,
  "source_csv": "path/to/final_blyat_v3.csv",
  "mode": "free_form",
  "free_form_text": "название заведения и особенности"
}
```

### Конкурентный анализ

```json
{
  "report_type": "competitive",
  "source_csv": "path/to/final_blyat_v3.csv",
  "reference_place": {
    "name": "Название заведения",
    "address": "адрес",
    "yandex_maps_link": "https://yandex.ru/maps/org/...",
    "menu_file": "/path/to/menu.pdf",
    "website": "https://..."
  }
}
```

- `menu_file` — путь к локальному PDF/PNG или поле `menu_url` для скачивания.
- Для 11 мест жёстко: 10 из БД + 1 из `reference_place`.

---

## Что упраздняем

- Режим **competitors** в текущем виде: отдельный тип отчёта «обзор конкурентов» без опорного заведения пользователя и без фиксированных 11 мест.
- Аргумент оркестратора **`--mode market|competitors`**: выбор типа отчёта только из входного запроса (`report_type`).

---

## Порядок внедрения (тактика)

1. **Контракт и оркестратор**
   - Ввести в `input_request.json` поле `report_type: "market" | "competitive"`.
   - Для обратной совместимости: если `report_type` нет, считать `"market"` при наличии `template` или `free_form_text`.
   - Оркестратор: читать `report_type` из входного JSON, передавать его в Block6 вместо `args.mode`; убрать `--mode` из argparse (или оставить только для переопределения в тестах).

2. **Block1**
   - Добавить ветку `report_type === "competitive"`: парсинг `reference_place`, поиск 10 мест в БД, сборка 11-й карточки, выдача 11 мест с пометкой `is_reference_place` для последнего.

3. **Block2**
   - Определять по `is_reference_place` и наличию `reference_place.menu_file` / `menu_url` использование пользовательского меню для 11-го места; для остальных — текущая логика.

4. **Block3**
   - Для места с `is_reference_place === true` и наличием `yandex_maps_link` извлекать org_id из ссылки и парсить отзывы только по нему; для остальных — как сейчас.

5. **Block4 / Block5**
   - Убедиться, что принимают 11 мест и для 11-го подставляют данные из `reference_place` где нужно.

6. **Block6**
   - Удалить вызов текущего `report_competitors` как отдельного режима.
   - Реализовать новый сценарий отчёта для `report_type === "competitive"`: те же данные (b1–b5), но структура и промпты заточены под «ваше заведение (11-е) vs 10 конкурентов» и рекомендации вокруг 11-го места.
   - Секции и заголовки отчёта задать под этот сценарий (например: «Ваше заведение», «Сравнение с конкурентами», «Рекомендации» и т.д.).

7. **Контракты и README**
   - Обновить примеры `input_request.json` и описание в README: два сценария (обзор рынка / конкурентный анализ), формат `reference_place`, фиксированные 11 мест для конкурентного анализа.

После этого тактика и структура будут соответствовать описанной логике: один выбор в начале, два чётких сценария, один тип отчёта по каждому сценарию, с конкурентным анализом, завязанным на 11-е место.
